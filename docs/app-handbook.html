Chapter 1What Fatwood is

One sentence: Fatwood (fatwood.io) turns the firehose of new arXiv papers into a ranked, personalized list of research an engineer could actually build — portfolio-project discovery, not paper search.

The user journey has four beats:

  1. Describe the goal in prose. "I have 3 years of backend experience and want to move into fintech" is a valid query. One cheap LLM call compiles it into a transparent, editable plan: concrete research topics, arXiv category filters, a date window — shown as chips the user can modify. Editing a chip re-runs the search for free.
  2. The corpus gets ranked. ~28,000 recent papers (growing nightly) are filtered by SQL, scored by meaning (local embeddings), scored by exact words (BM25), fused, and returned with a match percentage.
  3. Two wildcards are always injected — high-relevance papers deliberately far from the user's existing experience. This is a product guarantee, not a ranking outcome: a great project must never be hidden behind a skill you could learn in a weekend.
  4. Opt-in analysis. For papers the user selects, an LLM reads the paper against their profile and produces a feasibility read: what you'd learn, how long it takes, what it says on a resume, an extension idea. Cached forever per (paper × profile version) so it's paid for once.

It's a real multi-user product: open signup with email verification, $1 starter budgets metered to the token, bring-your-own API keys, admin tooling, and a measured search-quality loop that gates every ranking change.

Interview framing When asked "what did you build?", lead with the loop, not the stack: "A research-discovery product with a measured relevance pipeline — every ranking change has to beat the previous one on an offline evaluation harness before it ships, and the harness itself is calibrated against a stronger LLM judge." That sentence signals product thinking, IR literacy, and engineering rigor in one breath — then let them pull on whichever thread they want.

Chapter 2Architecture at ten thousand feet

Browser (React/TS SPA) │ same origin — no CORS anywhere ▼ ASP.NET Core API ──► PostgreSQL (EF Core, provider-portable) │ ├─ /api/* controllers, rate-limited │ ├─ /auth-proxy/* reverse proxy → Entra native-auth API │ ├─ wwwroot the built SPA (single container) │ └─ in-memory: embedding index (21k×384 vectors), BM25 index │ ├─ CLI modes (same binary): ingest | embed | analyze | enrich | eval ▼ Azure Container Apps ◄─ CD (GitHub Actions, OIDC — zero stored cloud secrets) ▲ Cloudflare (DNS, TLS, Bot Fight Mode, apex redirect) — free tier

The load-bearing decisions:

One container serves everything

The SPA is built in a multi-stage Dockerfile and served from the API's wwwroot with a fallback to index.html. The same image runs the web host and every CLI job — ingestion, embedding, evaluation — selected by argv in Program.cs. Why: a single origin kills CORS complexity permanently (this later paid off unexpectedly — see the auth proxy in Chapter 7); one image means CI/CD builds one artifact and the nightly ingest cron job runs the exact bits that were tested. Cost: the frontend and backend can't scale independently, and a frontend-only change redeploys the API. At this scale, irrelevant; at 100× scale you'd split them behind a CDN.

Strict layering, provider-portable persistence

Api → Infrastructure → Application → Domain. The domain has zero dependencies. UseNpgsql appears in exactly one file; there is no raw SQL, no jsonb, no PG-specific concurrency (xmin) — the ingestion lock uses a portable Guid concurrency token instead. Why: the stated goal was "SQL Server swap = package + one line + regenerated migrations," which is also a forcing function for clean data access. Cost: we gave up upserts via ON CONFLICT (see Chapter 3) and pgvector (see Chapter 4). Both costs were paid deliberately and both have documented escape hatches.

In-memory search indexes

All 21k embedding vectors (~32 MB) and the BM25 postings live in process memory, loaded at startup. Why: at this corpus size, brute-force cosine over the candidate set is single-digit milliseconds — an ANN index or pgvector would add operational surface for zero user-visible gain. The honest scaling answer: this design is comfortable to ~500k papers; past that you'd move vectors to pgvector with HNSW (accepting the PG lock-in, or an external vector store) and stream BM25 from an inverted index on disk. Knowing exactly where your design runs out is worth more in an interview than pretending it never does.

Interview moment — "why not microservices?" "Because the failure mode of this product was never load — it was quality and cost. A modular monolith with strict internal layering gave me the seams (Domain/Application/Infrastructure) without paying the distributed-systems tax: no service discovery, no partial failures between my own components, one deploy to reason about. The layering means extracting a service later is a refactor, not a rewrite."

Chapter 3Ingestion: 28k papers without getting banned

Query API over OAI-PMH

arXiv offers two access paths. OAI-PMH is the bulk-harvest protocol, but its "sets" are whole archives (cs, not cs.LG) — harvesting six sub-categories would mean over-fetching an order of magnitude and filtering client-side. The query API supports cat:cs.LG AND submittedDate:[X TO Y] ranges directly. For a 90-day × 6-category window (~30–60k records, 300–600 paged requests), that's well within polite-use territory at 1 request / 3 seconds. Documented threshold: multi-year or whole-archive harvests should switch to OAI-PMH.

Politeness is engineered, not hoped for

A singleton DelegatingHandler enforces ≥3s global spacing across all categories and pages (a semaphore plus last-request timestamp), and it sits inside the Polly retry pipeline — so retries are throttled too. Retries honor Retry-After, exponential backoff with jitter, 30s attempt timeouts.

Idempotency without ON CONFLICT

Re-running a backfill must change nothing. The upsert is provider-portable: load the page's existing papers into a dictionary by arxiv_id (unique index), then add-or-mutate in one SaveChanges per 100-entry page. A unique-violation race (two processes, same page) is handled by re-fetching the existing set once and re-applying — detected by re-query, not by sniffing SQLSTATE. Tradeoff: two round-trips where PG's ON CONFLICT would take one; ingestion is a background job, so latency is worthless there — portability wins.

Incremental sync via high-water marks

Each category persists the max submittedDate seen; the nightly delta queries [HWM → now] with ascending sort so the mark advances monotonically and pages stay stable mid-run. Boundary overlap is harmless because the upsert is idempotent — the two designs lean on each other. Known limitation, documented: keying on submission date misses revisions to already-fetched papers; lastUpdatedDate ranges are the drop-in fix when it matters.

Exactly one ingestion at a time — across processes

The scheduler runs in-process, but a CLI backfill is a separate process, so an in-memory semaphore can't serialize them. A single-row lease table carries a Guid stamped as an EF concurrency token: read → if free or stale (>120 min = crash recovery) → write holder + new stamp. Losing the optimistic-concurrency race means someone else got the lease. Why not PG advisory locks: provider portability, again — and the lease survives process death legibly (you can see who holds it and since when).

Interview moment — "how do you make a scheduled job safe?" This chapter is a complete answer: idempotent writes (re-runs are no-ops), persistent progress markers (crash mid-run, resume where you left off), a cross-process lease with staleness recovery (no zombie locks), and an audit table per run (papers fetched/added/updated, error text). Name all four; most candidates name one.

Chapter 4The search pipeline

Search is a staged pipeline where every stage is a config flag, every flag defaults to the measured winner, and flipping one is a deliberate human act recorded in configuration:

query ─► LLM compiles SearchPlan (the ONLY place a model sees the query; it never sees the corpus) interpretation │ anchor topics │ categories │ date window │ no-code flag │ HyDE abstract │ intent │ ▼ Stage 0 SQL filters (categories, date, has-code) ~28k → candidates Stage 1 Dense retrieval: bge-small-en-v1.5 (local ONNX, 384-dim) score = ½·(whole-intent sim) + ½·(best single anchor sim) anchors = each topic + the HyDE hypothetical abstract Stage 1b BM25 (k1=1.2, b=0.75) fused via Reciprocal Rank Fusion (k=60) Stage 2 Signal blend (recency/code/citations) — ALL WEIGHTS ZERO (measured) Stage 3 Cross-encoder rerank — OFF (measured, three times) Finally 2 wildcard slots: high-relevance papers LEAST similar to the user's experience (contractual, excluded from eval)

Why local embeddings

bge-small-en-v1.5 runs via ONNX Runtime in-process: no per-query API cost, no network latency, no vendor coupling for the highest-volume operation in the product. The critical operational detail: queries need the model's instruction prefix, documents don't, and vectors are keyed by model version — a model swap re-embeds the corpus (~45 min locally, nightly-self-healing batches in the cloud) and search degrades until it completes. That coupling is the real price of owning your embeddings.

Multi-anchor: the 0.38 disaster that taught us the shape of the problem

The compiler produces 8–15 anchor topics. Scoring each paper by its best topic match alone cratered nDCG from 0.55 to 0.38 — a paper nailing one narrow topic beat papers matching the whole intent. Pure whole-intent scoring dilutes multi-topic queries instead. The measured fix is the 50/50 blend above. Lesson worth quoting: the failure wasn't the idea, it was the max — any scoring term that lets one component single-handedly set a paper's score is a hijack channel. That exact mechanism came back a year… a day later with HyDE.

Hybrid retrieval: +17% and the reason both signals exist

Embeddings blur exact terminology ("DPO" and "RLHF" live close together); BM25 can't see that "chatbots making things up" means hallucination. Running both and fusing with RRF — rank-based, so no score-scale fighting — was the single biggest measured win: 0.523 → 0.614 nDCG@10. RRF's k=60 is the standard damping constant; we didn't tune it, deliberately: tuning a fusion constant on 21 queries is how you overfit an eval set.

HyDE: the shipped win, and the honest asterisk

The compiler writes the abstract of the hypothetical ideal paper for the query, and that text — embedded without the query prefix, because it's document-shaped — joins the anchor set. Abstracts match abstracts far better than topic lists do. Measured: +0.020 nDCG@10, with the gains exactly where predicted: queries phrased in user language rather than paper language ("how do people tell if text was written by an AI" +0.09) and vague career queries (+0.10). The asterisk: two acronym-dense queries got worse, because the abstract competes in the best-anchor max and can hijack it. We then measured three attempted fixes — skip HyDE for "precise" queries (lost: HyDE helps 17 of 21 precise queries), blend the vector instead of anchoring it (fixed the outliers, diluted everything else: a wash), and a weighted blend (worse than both). The outliers are an accepted, documented cost. This sequence — win, diagnose, three falsified fixes, documented acceptance — is the single best engineering story in the codebase.

The graveyard (negative results are load-bearing)

Wildcards: exploration as a contract

Two slots per search go to high-relevance papers least similar to the user's experience. They're excluded from eval scoring (they're serendipity, not a relevance claim) and can't be removed by any ranking change. Structurally protecting exploration also protects the future feedback loop: a ranker trained on clicks of what it chose to show eats its own tail; the wildcards guarantee off-distribution exposure forever.

Interview moment — "design a recommendation/search system" Walk this pipeline in order and you've answered the classic system-design prompt: query understanding (LLM compile), candidate generation (SQL + dense + lexical), fusion (RRF), ranking policy (measured weights), exploration (wildcards), and — the part most candidates skip — the evaluation loop that decides what ships. Then volunteer one negative result. Interviewers remember the candidate who said "we measured it and it lost."

Chapter 5Measurement: the part that makes everything else true

The prime directive of the repo: no ranking change ships without a measured delta. The machinery:

Ground truth as versioned artifacts

eval/queries.json: 40 frozen queries (authored to cover terminology mismatch, exact-term, cross-domain, constraint-heavy, vague-career, plus one adopted real query), each with a frozen compiled plan so scoring runs are deterministic and token-free. eval/judgments.json: ~4,900 LLM-graded (query, paper) pairs, 0–3, append-only across ranker generations (TREC-style pooling: every new ranker's head gets judged and accumulates, so recall estimates get more honest over time). Both live in git — every grade change is reviewable in a diff. Why files, not DB rows: ground truth is code-adjacent; it needs review, history, and blame more than it needs queries.

Metrics and the comparison protocol

nDCG@10 (position-discounted quality of the top of the list — the product metric), Recall@50 (is the good stuff findable at all), MRR (how fast the first great result appears). The protocol that keeps numbers honest: (1) implement the candidate behind a flag; (2) judge with the candidate active — new rankers surface unjudged papers, and an unjudged head scores as zero, so skipping this step silently biases against the candidate; (3) rescore both configs on the now-identical judgment set — never compare against a remembered number, because judgment growth deflates absolute scores; (4) winner takes the config default, loser gets documented so it's never re-litigated.

Judging the judge

All grades come from a haiku-class model (cheap enough to grade thousands of pairs), which raises the obvious question: can you trust it? eval calibrate answers it with data — a seeded, stratified 200-pair sample re-judged by sonnet, reporting exact agreement, within-one-grade agreement, and quadratic-weighted kappa. Result: QWK 0.83 on rubric v1, 0.81 on v2 — "excellent" agreement; disagreements are overwhelmingly adjacent-grade judgment calls that wash out in ranker-vs-ranker comparisons. Calibration also caught one systematic miss (the judge ignoring disqualifying constraints like "papers without public code"), which became rubric v2: a prompt rule, a version bump that hard-errors on mixed-rubric files, and a full 4,936-pair regrade that preserved pooling history. Absolute scores deflated under the stricter rubric (0.620 → 0.607 for the same config) — which is why the docs scream that cross-rubric comparisons are meaningless.

The CI gate: quality enforced by machinery

Every PR seeds a checked-in mini-corpus (all 4,069 judged papers, 2.5 MB gzipped, no vectors — CI re-embeds deterministically since embeddings are pure functions of model version), runs the frozen queries through the real ranker, and fails below an nDCG floor set just under the fixture baseline. The fixture tracks the full corpus almost exactly (0.618 vs 0.619) because judged papers dominate scoring. From that day on, "someone accidentally broke search" is a red X, not a mystery.

Online: telemetry and interleaving

Every product search logs its plan and per-rank results; bookmarks/analyses/not-interested join back to (search, rank). eval bias reports drift (category skew, position bias, wildcard yield, comfort-zone creep). And the endgame of the HyDE story runs in production right now: team-draft interleaving — each search silently interleaves the shipped stack with a HyDE-off candidate, tagging each slot's team; real clicks become votes. Offline said +0.02; the online ballot will confirm or veto it with user behavior. The improvement loop is deliberately never closed automatically — a human reads the reports and flips config — because a self-reinforcing click loop erodes the exploration guarantee first.

Interview moment — "how do you know your search is good?" Most candidates say "we looked at the results" or "A/B tests." You can say: "Three layers. Offline: 40 frozen queries, ~5k graded judgments, nDCG/Recall/MRR, with a strict judge-then-rescore-both protocol. The judge itself is calibrated — QWK 0.81 against a stronger model. Online: interleaving with click votes. And a CI gate that fails any PR dropping nDCG below a floor. Plus a graveyard of ideas that died in measurement, which is the system working."

Chapter 6The LLM layer and cost control

Three call sites, and only three

Call siteWhenModel tierWhy
Query compilerOnce per new searchhaiku-classStructured extraction + expansion; cheap, fast, good enough — and the plan is editable, so mistakes are user-correctable
Paper analysisOpt-in, per selected paperconfigurable per stepThe expensive, high-value read; cached forever per (paper × profile version)
Relevance judgeEval only, never in producthaiku-class (sonnet for calibration)Thousands of grades per run; cost scales with corpus

Design rules that matter: the model never sees the corpus (it compiles intent; retrieval is deterministic), every call uses structured outputs against a JSON schema (no parsing prose), and models are selected per step via configuration — swappable from the admin UI without a deploy. Executing a search plan costs zero tokens; only compilation and opt-in analysis spend money. That's why the plan chips exist: editing a filter re-runs retrieval free instead of paying for recompilation.

Budgets: a ledger, not a counter

Every user gets a $1 starter budget. Spend is recorded per LLM call in micro-dollars (integer arithmetic — no floating-point money) in an append-only usage table, with grants in a second table; the balance is a query, not a mutable column. Real token counts from API responses drive the metering, priced per model from the registry. Admins/Owners are unlimited by role. Why a ledger: auditability (every cent traces to a call), no lost-update races on a balance column, and grants/refunds are inserts, not arithmetic.

BYO keys, done paranoidly

Users can bring their own Anthropic key: encrypted at rest with ASP.NET Data Protection (keyring persisted in the DB so it survives container restarts), write-only from the client's perspective — the UI can set or clear a key but never read one back. BYO calls bypass the budget meter (their key, their bill).

Interview moment — "how do you control LLM costs in production?" "Architecturally, then financially. Architecturally: the expensive call is opt-in and cached forever; the frequent call is the cheapest model tier; plan edits re-execute deterministically for free. Financially: per-user budget ledgers in integer micro-dollars metered from real token counts, per-step model configuration so cost tuning is an ops change, and BYO keys to shift cost to power users. The mistake I designed against is the implicit one — a feature that spends tokens without anyone deciding it should."

Chapter 7Accounts, auth, and abuse resistance

Why Entra External ID, and why native auth

Identity is the classic buy-not-build: password storage, verification codes, token lifecycle, and the breach liability all belong to a specialist (Entra External ID's free tier covers 50k monthly active users). The interesting decision came after: the hosted sign-in page is brandable only within Microsoft's template — it looked like a Microsoft property and broke the product's theming (no dark mode, different fonts, a jarring redirect to ciamlogin.com and back). So sign-in moved into the app using the native authentication API: our own React screens drive the sign-up/sign-in/reset state machine, and the URL never leaves fatwood.io.

The catch that makes this a good war story: Entra's native-auth endpoints don't serve CORS headers — they assume mobile apps, not browsers. The fix leans on the single-origin architecture from Chapter 2: a small allowlisted reverse proxy at /auth-proxy forwards exactly the known auth paths (and nothing else) to the tenant, making the browser's requests same-origin. Two real-world potholes are pinned by tests: the SDK calls resetpassword/v1.0/* even though the docs say password_reset (both are allowlisted; a Theory test enumerates every path the SDK actually uses), and offline_access must be requested explicitly or sign-in fails with a cryptic missing-refresh-token error.

The branded email, and the three-layer debugging story

Verification codes originally arrived as generic Microsoft emails. The fix chained four services: Azure Communication Services sending from noreply@fatwood.io (SPF/DKIM verified), an Entra custom email provider extension calling our API on every OTP send, our endpoint rendering the branded template, and Entra falling back to its stock email if we return non-200. It failed silently at three layers, each caught with evidence, not guessing: (1) the event listener creation had silently failed — a script printed DONE over an HTTP 400 (Graph rejects includeAllApplications for this listener type); (2) the fallback behavior lives on the extension, not the listener; and (3) the smoking gun — zero hits in origin logs, so the callout never arrived: Cloudflare Bot Fight Mode was issuing a managed challenge to Microsoft's webhook (proven from firewall event logs, ASN 8075, src=botFight). Rather than weaken the bot protection site-wide, the extension was re-pointed at the container app's direct origin URL, bypassing Cloudflare for that one server-to-server call — which required teaching the app registration a second identifier URI, because Graph validates that the auth audience domain matches the target URL domain.

Authorization: three tiers, server-enforced

"Make my buddy an admin" shouldn't hand over the LLM config. Roles are Member (normal user), Admin (people ops: accounts, budget grants, invite codes — plus an unlimited personal budget), and Owner (everything: role changes, global model settings, ingestion, bulk analysis). Policies are enforced on controllers server-side; the UI merely reflects them. Role changes are Owner-only and self-role-change is blocked. A data migration promoted the pre-split admins (all of whom were the operator) to Owner.

Abuse resistance, layered

Rate-limit policies per surface (auth endpoints, LLM endpoints, global); email verification before activity; a feature-flagged invite-code gate that can flip signup from open to invite-only with a config change (the "cake and eat it too" requirement — open now, gateable under attack); $1 budgets bounding worst-case burn per account; and Cloudflare's bot layer in front of everything. Known-and-accepted gap, documented: disposable-email farming — a blocklist, per-IP caps, and Turnstile are designed but deliberately deferred until there's evidence of need.

Interview moment — "tell me about a hard bug" The OTP story has the ideal shape: symptom (stock email despite a configured custom provider), three false floors (silent 400 under a success message; config property on the wrong resource; and an invisible network-layer block), the evidence habit (origin request logs proved the callout never arrived; Cloudflare firewall analytics named the blocker), and a judgment call (don't disable bot protection globally for one webhook — route around it and keep the security posture). Tell it in that order.

Chapter 10The interview playbook

The 90-second project summary (memorize the beats, not the words)

"Fatwood is a research-discovery product I run in production — it turns new arXiv papers into ranked, personalized portfolio-project candidates. An LLM compiles the user's prose into an editable search plan, then a deterministic hybrid retrieval stack ranks ~28k papers: local ONNX embeddings with multi-anchor scoring plus HyDE, fused with BM25 by reciprocal rank fusion. The part I'm proudest of is the quality loop: 40 frozen eval queries, about five thousand graded judgments from a calibrated LLM judge, a strict comparison protocol, a CI gate that fails any PR that drops nDCG, and live interleaving in production right now to confirm the latest win with real clicks. Several 'obviously good' ideas — rerankers, citation boosts — died in measurement, and the negative results are documented so they never get re-litigated. It's a full multi-user product: Entra native auth served from my own UI through a same-origin proxy, per-user token-metered budgets in a micro-dollar ledger, three-tier authorization, IaC with secretless OIDC deploys."

Chapter 11The AI story — and where you provided the value

You'll be asked, directly or sideways: "How much of this did AI write?" The wrong answers are pretending it wrote none of it, or shrugging that it wrote all of it. The right answer is that you ran an AI-assisted engineering process with the same accountability structure any senior engineer applies to any leverage — a team, a library, a contractor — and you can prove it, because the process left artifacts.

An honest division of labor

How to say it in an interview

"I built it AI-assisted, deliberately, and I'd describe my role as the engineering lead on a very fast team: I set direction, made the tradeoff calls, reviewed everything, and owned verification. The guardrails were real — every ranking change had to beat baseline on an offline eval harness before shipping, the LLM judge behind that harness is itself calibrated against a stronger model, CI has 113 tests plus an nDCG regression gate, and I was the human in the loop on every production deploy and every live debugging session. I can walk you through any layer of it and defend why it's built the way it is — that's what this conversation is."

Then demonstrate — pick the layer they know best and go deep. The credibility comes from fluency, not from claiming keystrokes.

Why this is a strength, not an asterisk

Teams are actively hiring for engineers who can direct AI leverage without losing rigor. This project is evidence of exactly that skill: scope control (phases with acceptance criteria), verification culture (measured deltas, calibrated judges, gates), operational ownership (you ran the incident debugging, you hold the keys), and honest documentation (negative results recorded so they're never repeated). The fact that some of the code was typed by a model is the least interesting thing about it — what matters is that the system producing the code had you as its quality function.
